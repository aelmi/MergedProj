Architecture review

SQL Server Index Maintenance — v1.3

A configuration-driven redesign of weekly maintenance, plus a new daily safety-net job

Status: Draft for management review Author: DBA Team Date: 20 May 2026
TL;DR

We are replacing the wrapper around our SQL Server index maintenance with a configuration-driven version, adding a daily statistics-refresh job, and fixing four silent defects that have been making weekly maintenance run for 2 to 5 days. The work engine itself — Ola Hallengren's industry-standard IndexOptimize procedure — is unchanged. The change is fully reversible.

What changed

Five concrete differences between today's setup and v1.3.

What each row means

Weekly run time
The job that keeps our databases healthy currently runs for 2 to 5 full days and overlaps with business hours. The new version is expected to complete in hours.
Known issues
Four silent bugs in the current wrapper script. They mean rebuilds are cancelled after 30 seconds, the wrong technique is sometimes used, and the underlying tool is prevented from running efficiently. The "4 defects" tab explains each one in plain English.
Parameter changes
Today, adjusting any maintenance setting means editing a PowerShell script and redeploying it across the fleet. With v1.3, settings live in a single XML file that anyone with config access can adjust without touching code.
Weekday coverage
Monday to Saturday currently has zero maintenance. The new daily job refreshes statistics every weeknight and rebuilds any index that has become severely fragmented, keeping the database performing well between weekends.
Scalability
The current setup processes one database at a time. The new architecture is designed so that adding more parallel workers — to use the full server — is a deployment-only change in a future release, with no script rewrite needed.

How it works

A four-step flow. Same engine as today; new plumbing around it.

The key thing to convey is that we are not building new technology — we are putting proper engineering discipline around well-established tooling. The work engine in step 3 is Ola Hallengren's IndexOptimize, which is the de facto industry standard for SQL Server maintenance and is in production at thousands of organisations. What this release adds is the missing scaffolding around it.

The four silent defects in the current wrapper

Each one has been quietly causing problems. Together they explain most of why the weekly job runs for days instead of hours.

Defect 1 — The 30-second timeout

Severity: high • Impact: most rebuilds silently cancelled
In plain English

When rebuilding a large index, the script gives up after 30 seconds. The database engine keeps working, but the script has already moved on, and treats the job as successful. Indexes that need an hour of rebuild work get left in a partially-fixed state every week, which is why fragmentation has been creeping up despite the weekly job running to completion.

Business impact

Large tables never get properly maintained. Query plans on those tables choose progressively worse strategies, which shows up as slow reports and slow application screens.

Technical root cause

The line $MaintenanceCommand.CommandTimeout = $TimeOut references a variable named $TimeOut that is never defined in the script (the parameter is called $TimeLimit). The undefined value resolves to zero in PowerShell, but the .NET SqlCommand object treats that as "use the default", which is 30 seconds. In v1.3 we set it explicitly to 0, which means no client-side timeout — the database engine controls duration through its own time-limit parameter.

Defect 2 — The wrong rebuild technique

Severity: high • Impact: hours wasted on operations that cannot finish
In plain English

SQL Server offers two main ways to fix a fragmented index: a "gentle" approach called REORG, and a "comprehensive" approach called REBUILD. The current script prefers REORG for moderately-fragmented indexes because it sounds safer. Unfortunately REORG is single-threaded and on a large, busy index it can run for eight hours without finishing — meanwhile REBUILD would have done the same work in 20 minutes. We have seen this happen in production.

Business impact

A single problematic large index can hold up the entire weekly maintenance window. It also cannot update statistics, so a separate statistics-update pass is required afterwards — doubling the work for no benefit.

Technical root cause

The variable $FragmentationMedium is hardcoded to "INDEX_REORGANIZE,INDEX_REBUILD_ONLINE,INDEX_REBUILD_OFFLINE". IndexOptimize takes the first applicable action, so REORG always wins for the 5-to-30-percent fragmentation band. In v1.3, REORG is removed from both the Medium and High bands. Both now go straight to online or offline rebuild, which uses multiple CPUs, completes deterministically, and refreshes statistics as a side effect.

Defect 3 — The empty-configuration bug

Severity: medium • Impact: latent — would surface on first XML-driven run
In plain English

When a configuration setting is intentionally left blank — meaning "use the default" — the current code passes a blank string to the database engine. The engine refuses to accept a blank string for these settings and fails the entire job. This was a latent bug that would have shown up immediately once we adopted XML-based configuration.

Business impact

Without this fix, the very first run of the new configuration system would have failed across the fleet. The fix makes the XML "blank means default" behaviour work the way an operator would expect.

Technical root cause

An empty XML value="" attribute is read into PowerShell as an empty string "". The .NET SqlCommand.Parameters.AddWithValue call then passes "" to the stored procedure rather than NULL. Ola's validation logic parses the empty string into a single empty action token and raises a "value not supported" error. The v1.3 wrapper has a new helper called Convert-XmlValue that maps any whitespace-or-empty XML value to System.DBNull.Value, which the procedure receives as proper SQL NULL and treats as "apply the default".

Defect 4 — The database-by-database loop

Severity: high • Impact: prevents efficient ordering and time-limit features
In plain English

The current script processes databases one at a time in a PowerShell loop. The tool we use — Ola Hallengren's IndexOptimize — has features for "work on the largest databases first" and "stop everything cleanly if we have been running for X hours", but those features only work when the script lets the tool see all the databases at once. By looping in PowerShell, we are preventing the tool from doing its job efficiently.

Business impact

If maintenance is running out of time, the current script keeps starting work on small databases at the end of the alphabetical list. The new approach starts with the largest databases first and stops cleanly if the time budget runs out — so the most important work always gets done.

Technical root cause

The original wrapper does foreach ($db in $dbs) { IndexOptimize @Databases = $db }. Each invocation sees only one database, so @DatabaseOrder and @TimeLimit are meaningless. In v1.3, a single call passes @Databases = 'USER_DATABASES, -ExcludedDB1, -ExcludedDB2'. The procedure enumerates databases internally, applies ordering, propagates the time limit across the run, and natively skips non-primary Availability Group replicas.

Net effect. All four defects are silent — none of them cause the job to report failure. They produce slow, incomplete maintenance and tie up the maintenance window. Fixing them is the single largest contributor to the expected reduction from days to hours.

Architecture in detail

Seven layers from the SQL Agent trigger down to the audit log.

Components

Weekly DBA Jobs modified

Existing SQL Agent job. Only the Step 1 command changes — adds -JobType Weekly. Schedule (Sun 04:00) and Step 2 (Cycle Error Log) remain identical.

Daily DBA Jobs new

New SQL Agent job. Two steps — maintenance via PowerShell and Cycle Error Log. Schedule Mon-Sat 18:00. Completely independent from the existing weekly job.

InstanceUpdateStats.ps1 modified

The PowerShell wrapper, rebuilt to v1.3. Reads XML configuration, computes MaxDOP from CPU detection, calls IndexOptimize once with the full database list, fixes all four defects.

IndexOptimizeConfig.xml new

Fleet-wide configuration file. Two default profiles (Weekly, Daily) and an optional InstanceOverrides section for edition-licensed CPU caps or other per-instance exceptions.

XMLFunctions.ps1 modified

Existing helper library. Two existing functions unchanged. One new function (fnReadIndexOptimizeParams) appended that reads the new XML and merges defaults with overrides.

SQLExclusions.xml unchanged

Existing per-instance exclusion list. Reused as-is. The new wrapper translates the exclusions into the stored procedure's -DatabaseName syntax.

dbo.IndexOptimize unchanged

Ola Hallengren's stored procedure. The work engine. Same version in production today; no upgrade needed.

dbo.Queue, dbo.QueueDatabase new

Two new tables in MaintenanceDB. Required when @DatabasesInParallel = 'Y' is set, even with a single consumer. Installed now for v1.4 forward compatibility.

dbo.CommandLog unchanged

Existing audit table. One row per index or statistics command. Already populated by the current wrapper; v1.3 keeps logging to it.

Configuration model

One XML file controls every parameter for both profiles, on every instance in the fleet:

How parameters are resolved at run time

  1. The wrapper looks up the Defaults/Job matching the current -JobType argument.
  2. It loads every parameter into a hashtable.
  3. If an InstanceOverrides/Instance block matches the current server name and contains a Job matching the current JobType, only the parameters listed in that override block are replaced in the hashtable.
  4. The resulting hashtable is passed to IndexOptimize as ADO.NET parameters, with empty values converted to SQL NULL.

Deployment plan

Six files, eight steps per instance. Every step has a verification check and an explicit rollback path.

Files in this release

Status File Purpose
New IndexOptimizeConfig.xml Fleet-wide configuration with Weekly + Daily profiles
Modify InstanceUpdateStats.ps1 Wrapper rewritten to v1.3. Original structure preserved, every change marked with v1.3 comment
Modify XMLFunctions.ps1 One new function appended. Existing two functions unchanged
No change DBAFunctions.ps1 Pure loader; XMLFunctions already in load list
New 00_Install_QueueTables.sql Installs Queue and QueueDatabase tables in MaintenanceDB
Update 01_Update_WeeklyDBAJobs_Step.sql Idempotent script adding -JobType Weekly to the existing Step 1 command
New 02_Create_DailyDBAJobs.sql Creates the new Daily DBA Jobs SQL Agent job
No change SQLExclusions.xml Per-instance exclusions, consumed as today

Deployment sequence (per instance)

  1. Capture baseline Record current weekly duration from dbo.CommandLog. Median run time becomes the success metric.
  2. Install queue tables Run 00_Install_QueueTables.sql against the MaintenanceDB. Verify all five required objects are present.
  3. Deploy file changes via source control Commit the three updated files. Push, review, merge. DSC distributes them to the host.
  4. Smoke test the wrapper Run the wrapper manually against one small test database. Confirm Ok return and new rows in dbo.CommandLog.
  5. Update the Weekly job's step command Run 01_Update_WeeklyDBAJobs_Step.sql. Adds -JobType Weekly to the existing command line.
  6. Create the new Daily job Run 02_Create_DailyDBAJobs.sql. The script refuses to recreate if the job already exists.
  7. Manual first run of Daily DBA Jobs EXEC msdb.dbo.sp_start_job against the new job. Confirm run_status = 1 in sysjobhistory before letting the schedule fire on Monday.
  8. Monitor first scheduled runs Watch Monday 18:00 (first Daily on schedule) and the following Sunday 04:00 (first new-style Weekly). Compare duration against baseline from Step 1.

Rollback

Step 1: Disable the new Daily DBA Jobs (one line).

Step 2: Revert the Weekly DBA Jobs Step 1 command to the original (one sp_update_jobstep call; the original command is documented in the rollback block of 01_Update_WeeklyDBAJobs_Step.sql).

Step 3: If a defect is found in the new files, revert the commit in source control. DSC restores the previous versions on the next consistency pass. Do not edit files directly on the host — DSC will overwrite any local change.

Step 4: Leave the new Queue tables in place. They are empty when not in use and have no operational cost.

Why this is safe

Every step is reversible in seconds. The existing Weekly job's structure is preserved — only its command line gets one extra argument, which can be removed with a single SQL call. The new Daily job is brand new and isolated from everything else, so disabling it has no side effects. The work engine — Ola Hallengren's stored procedure — is the same one already in production. The configuration files live in source control, with DSC enforcing the desired state on each host.
